# Voting System API

REST service for registering votes across multiple independent elections. Voters can be
registered, blocked, and unblocked. Each election owns its voting options, and a voter may cast
at most one vote per election. Data is persisted in PostgreSQL.

## Overview

The service exposes a small set of REST endpoints for managing voters, elections, options, and
votes. It enforces four business invariants:

- INV-1: a voter casts at most one vote per election.
- INV-2: a blocked voter cannot vote.
- INV-3: a vote must reference an option that belongs to the target election.
- INV-4: voter identity (name) is unique.

The one-vote-per-election rule is guarded both in the domain and by a composite unique constraint
on the `votes` table, which is the final line of defense under concurrent requests.

## Architecture

The project follows a hexagonal (ports and adapters) design split into four Maven modules with
compile-time dependency enforcement. The dependency direction is strictly from outer to inner.

| Module   | Artifact       | Spring | Depends on        | Responsibility                                   |
|----------|----------------|--------|-------------------|--------------------------------------------------|
| core     | domain         | no     | none              | Aggregates, value objects, domain exceptions     |
| usecases | application    | no     | core              | Inbound (`ApiPort`) and outbound (`SpiPort`) ports, application services |
| adapters | infrastructure | yes    | usecases, core    | Web controllers, JPA persistence, configuration, cross-cutting concerns |
| app      | bootstrap      | yes    | adapters          | Spring Boot entry point, configuration resources, database migrations |

`core` and `usecases` are framework-free. Application services are plain constructor-injected
objects; their Spring wiring lives in `adapters.config.UseCaseConfig`, keeping the inner modules
independent of any framework.

Package layout uses the `clean_hex` variant:

- Domain: `core.model`, `core.exception`, `core.service`
- Application: `usecases.api` (inbound ports), `usecases.spi` (outbound ports), `usecases.service`, `usecases.exception`
- Infrastructure: `adapters.web`, `adapters.web.dto`, `adapters.web.mapper`, `adapters.web.error`, `adapters.web.ratelimit`, `adapters.web.security`, `adapters.persistence.jpa`, `adapters.health`, `adapters.metrics`, `adapters.config`
- Bootstrap: `app`

## Prerequisites

- JDK 17
- Maven 3.9 or newer (a local installation; the project does not ship a wrapper)
- Docker and Docker Compose (for the containerized stack and integration tests)

## Run

### With Docker Compose

The full stack (application, PostgreSQL 16, Redis 7) starts with a single command:

```bash
docker compose up --build
```

The API becomes available at `http://localhost:8080` once the health checks report ready.

## Test

```bash
mvn test
```

## API Documentation

OpenAPI documentation is generated by SpringDoc.

- Swagger UI: `http://localhost:8080/swagger-ui.html`
- OpenAPI document: `http://localhost:8080/v3/api-docs`

Operational endpoints:

- `http://localhost:8080/actuator/health`
- `http://localhost:8080/actuator/metrics`
- `http://localhost:8080/actuator/prometheus`

### Endpoints

| Method | Path                                  | Success | Description                     |
|--------|---------------------------------------|---------|---------------------------------|
| POST   | `/api/voters`                         | 201     | Register a voter                |
| GET    | `/api/voters/{voterId}`               | 200     | Get a voter                     |
| POST   | `/api/voters/{voterId}/block`         | 200     | Block a voter                   |
| POST   | `/api/voters/{voterId}/unblock`       | 200     | Unblock a voter                 |
| POST   | `/api/elections`                      | 201     | Create an election              |
| GET    | `/api/elections/{electionId}`         | 200     | Get an election with options    |
| POST   | `/api/elections/{electionId}/options` | 201     | Add an option to an election    |
| POST   | `/api/elections/{electionId}/votes`   | 201     | Cast a vote in an election      |

## API Examples

The following examples were verified against a running stack (`docker compose up --build`).
The base URL is `http://localhost:8080`, all request and response bodies are JSON, and
identifiers are UUIDs.

### Health Check

```bash
curl -s http://localhost:8080/actuator/health
# Response: {"status":"UP"}
```

### Register a Voter

```bash
curl -s -X POST http://localhost:8080/api/voters \
  -H 'Content-Type: application/json' \
  -d '{"name":"Jane Doe"}'
# Response: {"id":"<uuid>","name":"Jane Doe","status":"ACTIVE"}
```

### Get a Voter

```bash
curl -s http://localhost:8080/api/voters/<voterId>
```

### Block a Voter

```bash
curl -s -X POST http://localhost:8080/api/voters/<voterId>/block
# Response: {"id":"<uuid>","name":"Jane Doe","status":"BLOCKED"}
```

### Unblock a Voter

```bash
curl -s -X POST http://localhost:8080/api/voters/<voterId>/unblock
# Response: {"id":"<uuid>","name":"Jane Doe","status":"ACTIVE"}
```

### Create an Election

```bash
curl -s -X POST http://localhost:8080/api/elections \
  -H 'Content-Type: application/json' \
  -d '{"title":"Mayor Election 2025"}'
# Response: {"id":"<uuid>","title":"Mayor Election 2025","options":[]}
```

### Add an Option to an Election

```bash
curl -s -X POST http://localhost:8080/api/elections/<electionId>/options \
  -H 'Content-Type: application/json' \
  -d '{"label":"Candidate A"}'
# Response: {"id":"<electionId>","title":"...","options":[{"id":"<optionId>","label":"Candidate A"}]}
```

### Get an Election with Options

```bash
curl -s http://localhost:8080/api/elections/<electionId>
```

### Cast a Vote

```bash
curl -s -X POST http://localhost:8080/api/elections/<electionId>/votes \
  -H 'Content-Type: application/json' \
  -d '{"voterId":"<voterId>","optionId":"<optionId>"}'
# Response: {"id":"<uuid>","electionId":"...","voterId":"...","optionId":"..."}
```
